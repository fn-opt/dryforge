# dryforge

*[English](./README.md)*

> Ready or Set, and Go!
> 의도와 다르게 써진 문서든, 머릿속 아이디어든. 명령어 두 개로 끝.

**설치:**

```
/plugin marketplace add fn-opt/dryforge
/plugin install dryforge
```

---

## 이것이 dryforge가 만들어진 이유입니다.

> **에이전트의 탈선을 막습니다.**

- 뭘 만들지 파악도 안 하고 코딩부터 합니다.
- spec을 써도 의도와 다른 방향으로 흘러갑니다.
- plan에 코드를 잔뜩 써놓는데, 실제 코드와 달라서 에이전트가 자기 문서랑 싸웁니다.

> **작업 효율을 근본적으로 개선합니다.**

- 설정 파일 하나 고치는 것도 테스트부터 잔뜩, 한 줄 바꾸는 것도 검증 세 번.
- 동시에 돌릴 수 있는 작업이 순서대로 하나씩 돌아갑니다.

### dryforge는 이 둘을 같이 고칩니다

문서의 역할을 바로잡고, 실제 코드에 맞춰 검증한 다음, 작업에 맞는 검증과 병렬 실행으로 만듭니다.

**`set → go`** — 문서를 뜯어고칩니다. 틀린 건 바로잡고, 빠진 건 채우고, 모호한 건 질문해요.

**`ready → go`** — 깊게 묻고, 문서를 직접 쓰고, 실행합니다. 처음부터 끝까지.

---

## dryforge의 차별화된 특징, /set

작업을 시작할 때, 문서가 아예 없는 경우는 드뭅니다. 대충이라도 뭔가 적어놓은 게 있어요 — 직접 쓴 spec이든, 다른 도구가 뽑아준 plan이든, 이전 세션에서 에이전트가 만든 거든.

문제는 그 문서가 맞는지 아무도 확인 안 한다는 겁니다. 틀린 경로, 빠진 요구사항, 코드랑 안 맞는 구조 — 실행 전에 잡으면 수정이고, 실행 후에 잡으면 재작업입니다.

다른 플러그인은 그 문서를 받은 그대로 실행합니다. `set`은 실제 코드베이스에 대고 문서를 해부합니다 — 본인도 모르게 틀어진 부분을 찾아내고, 바로잡고, 의도에 맞는 문서로 되살려서 엉뚱한 방향으로 흘러가는 걸 막습니다.

재작업이 줄고, 결과물의 품질이 올라갑니다.

---

## 어디서 시작하든, 결과는 같습니다

### 경로 1 — 문서가 이미 있을 때

```
/dryforge:set    →    /dryforge:go
```

`set`은 문서를 받아서 바로 실행하지 않습니다. 실제 코드베이스를 읽고 먼저 검증합니다:

- 코드랑 안 맞는 경로, 존재하지 않는 파일, 틀린 구조를 잡아냅니다.
- spec↔plan 양방향 추적 — 빠진 요구사항, 근거 없는 작업을 찾습니다.
- 섞여 들어간 구현 코드를 걷어내고, 기능 목표만 남깁니다.
- 모호한 건 추측하지 않고 물어봅니다.
- 작업 간 의존성을 계산해서 병렬 실행 순서를 잡습니다.

### 경로 2 — 아이디어에서 시작

```
/dryforge:ready    →    /dryforge:go
```

`ready`가 대화로 의도를 끌어냅니다. 체크리스트가 아니라 진짜 대화 — 기능 의도는 깊게 파고, 기술 선택은 추천안부터 던져서 빠르게 수렴합니다. 코드 보면 알 수 있는 건 알아서 처리하고, 사용자만 결정할 수 있는 것만 물어봅니다.

코드를 읽고, 프로젝트 맥락에 맞는 문서를 직접 씁니다. 빠진 결정이 없는지 자체 검증까지 마칩니다.

### `/dryforge:go` — 실행

새 세션에서 (`/clear`) 실행합니다.

- **의존성 그래프대로 병렬 실행.** 동시에 돌릴 수 있는 건 동시에. 최대 8개.
- **격리된 환경.** 각 작업은 독립된 git worktree. 파일 충돌 없음.
- **작업에 맞는 검증.** 작업마다 spec 리뷰, 묶음마다 통합 검증과 코드 리뷰.
- **막히면 물어봄.** 추측 대신 사용자에게 올림.
- **main 보호.** 승인 전까지 main은 건드리지 않음.

## 명령어

| 명령어 | 하는 일 |
|---|---|
| `/dryforge:ready <goal>` | 대화로 의도 파악 → 코드베이스에 맞는 문서 직접 작성 |
| `/dryforge:set <spec> <plan>` | 기존 문서를 실제 코드에 맞춰 검증·수정·완성 |
| `/dryforge:go` | 병렬 실행, 작업별 검증, 막히면 물어봄 |
